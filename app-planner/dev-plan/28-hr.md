# 28 - Human Resources Module

## Goal
Manage staff profiles, positions, assignments, availability, leave, shifts, rosters, shift swaps, payroll runs, and workforce administration while feeding role, department, and scheduling context used by clinical and operational modules.

## Source of Truth
- `app-write-up.md`, `opd-flow.md`, and `ipd-flow.md` are the single product/flow source of truth for this implementation plan; backend/frontend planner files and rules are alignment references only.
- Use `app-write-up.md` for HR scope.
- Use `01-policy.md` for role access, reusable UI, simple forms, responsive behavior, reports, and partial state refresh.
- Use access-control work from earlier dev-plan steps instead of redefining roles or permissions.

Use these route families only after confirming they exist in the current backend router/API contract. If a listed route is absent, record it as a backend gap and do not create a frontend-only endpoint, fake status, or local-only workflow.

## Implementation Scope
1. Staff directory with search, filters, status, department, role, position, and facility assignment.
2. Staff detail with profile, assignments, roles reference, availability, leave, shifts, roster, and payroll visibility where permitted.
3. Modal actions for add/edit staff profile, assign position, assign department/unit, record availability, request/approve leave, assign shift, swap shift, and run payroll actions where permitted.
4. Shift and roster views for nursing and department scheduling.
5. Reports for staff list, roster, leave, and payroll where permitted.

## UI and Workflow Contract
| Area | Requirement |
| --- | --- |
| Main screen | Use searchable staff list with compact filters. Do not overload with payroll and roster details in one list. |
| Staff detail | Use sections/tabs for profile, assignments, roles, availability, leave, shifts, and payroll. |
| Actions | Use modals for create/edit/approve/assign actions. Use focused pages only for complex payroll run review. |
| Integration | Staff assignments should be consumed by OPD/IPD/nursing/theater/HR scheduling without duplicating staff data. |
| Responsiveness | Mobile uses list plus focused detail; desktop supports list plus detail panel. |

## Reusable Components and Sync Contract
- Reuse `10-workspace-ui.md` workspace layout, shared form fields, shared modal/dialog shell, responsive detail panels, status badges, search/filter/table/list controls, async state views, and permission-gated action patterns before adding module-specific widgets.
- Use or create shared components for: staff profile card, staff search/list, assignment form, leave request modal, shift/roster table, availability selector, payroll run card, and permission-aware staff action menu.
- Keep common form layout, field behavior, validation-error display, server-error mapping, loading state, disabled state, and duplicate-submit protection shared; keep module-specific validation and submit mapping in feature controllers/repositories.
- Keep modal actions focused and return users to the same worklist/detail context after success; refresh only backend-backed affected rows, badges, panels, queues, counters, report previews, or form sections.
- Backend/frontend sync required: staff profiles, assignments, shifts, rosters, leave, availability, payroll, department/facility scope, notifications, reports, and audit records must stay synchronized.
- Do not create duplicate patient, encounter, admission, order, invoice, payment, report, notification, status, or action components when an existing shared pattern can represent the same job.

## Flow Synchronization Rules
- HR must not redefine application permissions; it should display or link to existing access-control assignments where needed.
- Department, unit, role, and availability changes should update relevant module filters, rosters, and assignment lists.
- Nurse roster changes should update nursing queues where backend supports it.
- Payroll data must remain restricted to permitted users.

## Access and State Rules
- HR admins manage staff records according to tenant/facility scope.
- Managers may approve leave/rosters only where permitted.
- Payroll access is separately guarded.
- After mutation, refresh only the affected staff row, roster entry, leave request, shift assignment, or payroll row.

## Reports and Printing
Staff profile, roster, leave summary, attendance/availability summary, payroll summary, and HR audit reports must use generated report templates from `35-reports-audit.md`.

## Concrete Implementation Contract
| Slice | Required implementation |
| --- | --- |
| Worklist/list data | Use `AppWorkspace` + `AppPaginatedListTable<StaffWorkItem>` with staff number/name, role/position, department/unit, assignment, shift/availability/leave status, and next action. Use `AppSearchBar` for staff, department, role, shift, status, and date filters. |
| Detail/display | Use staff detail with profile, positions, assignments, availability, shifts, leave, roster, payroll links, and audit activity. |
| CRUD/UI actions | Use `AppDialog` for create/edit staff profile, assignment, availability, leave request/approval, shift swap, roster publish/approve, and short payroll actions; use full page only for complex payroll/run review. |
| RBAC/ABAC | Gate with HR, unit, roster, payroll, and admin permissions plus tenant/facility/department scope and backend authorization. |
| Partial refresh | After HR mutation update only staff row, roster/day cell, leave badge, payroll summary, affected unit/department count, and notifications. |

Implementation must reuse `AppWorkspace`, `AppListTable`/`AppPaginatedListTable`, `AppSearchBar`/`AppListTableSearch`, `AppDialog`, shared form fields, `AppStateView`/`AsyncStateScaffold`, and access gates before adding feature-local UI. Do not reload the full workspace after modal actions.


## Done Criteria
- HR screens are simple, searchable, and role-aware.
- Staff/roster data supports clinical and operational modules without duplication.
- Payroll and sensitive data are restricted.
- UI remains consistent and responsive.
