# 35 - Reports, Dashboards, Print Templates, and Audit

## Goal
Provide role-aware dashboards, report definitions, report runs, scheduled reports, exports, audit logs, PHI access logs, data processing logs, compliance evidence, and a shared professional report/print template used by all modules.

## Source of Truth
- Use `app-write-up.md` for reports, dashboards, and audit scope.
- Use `opd-flow.md` and `ipd-flow.md` for patient-flow reports, encounter/admission summaries, service movement, billing, discharge, and queue turnaround metrics.
- Use `01-policy.md` for generated printing, permission-gated reports, simple UI, and partial state refresh.

## Implementation Scope
1. Role-aware dashboard landing pages with simple summary cards and links back to module queues.
2. Reports workspace with searchable report catalog, quick run, saved filters, schedules, exports, and history.
3. Shared print/report renderer for patient, clinical, billing, lab, radiology, pharmacy, discharge, operational, and administrative documents.
4. Audit log search/filter with clear event, user, module, action, record, timestamp, and facility context.
5. PHI access review and compliance evidence panels where permitted.
6. Turnaround-time reports for OPD, triage, lab, radiology, pharmacy, billing, IPD admission, discharge, bed turnover, and housekeeping where data is available.

## Shared Report/Print Template Contract
Printed output must be generated from data, not from the visible UI.

| Template Area | Required Content |
| --- | --- |
| Facility header | Logo, facility name, branch/facility name where relevant, address, phone, email, website, and optional regulatory/tax/license details where configured. |
| Document title | Clear report name, document/reference number, generated date/time, and generated by. |
| Patient context | Patient name, patient number, age/sex where available, encounter/admission number, department/unit, ward/bed where relevant. |
| Body | Clean sections/tables generated from backend data. Support page breaks, long notes, long item lists, and multi-page reports. |
| Footer | Page number, confidentiality/medical record note where appropriate, facility contact, and generated timestamp. |
| Signature block | Doctor, pharmacist, cashier, lab/radiology verifier, nurse, admin, or other role signature/approval lines where appropriate. |
| Permissions | Preview, print, export, download, and schedule actions must be permission-gated and audited. |

## Standard Report Types
| Area | Example Outputs |
| --- | --- |
| OPD/Clinical | Consultation summary, referral letter, prescription, patient report, follow-up note. |
| IPD/Discharge | Admission slip, inpatient summary, transfer note, discharge summary, discharge instructions. |
| Lab/Radiology | Lab result, sample summary, radiology report, imaging report reference. |
| Pharmacy | Prescription printout, medication label, dispensing receipt, discharge medicines list. |
| Billing/Claims | Invoice, receipt, refund note, statement, shift close, day close, claim/pre-authorization documents. |
| Operations | Bed occupancy, housekeeping, maintenance, biomedical, HR, subscription, audit, and compliance reports. |

## UI and Workflow Contract
| Area | Requirement |
| --- | --- |
| Dashboard | Summarize; do not replace module workflows. Cards should link to the exact module queue. |
| Report catalog | Use plain names and descriptions. Avoid technical query names in the UI. |
| Report run | Use a modal or focused panel for filters, date range, facility, department, status, and output type. |
| Preview | Preview generated report content before print/export when possible. |
| Exports | Export only data permitted by role and tenant/facility scope. |
| Responsiveness | Mobile supports report selection and preview; desktop supports catalog plus preview/detail area. |

## Flow Synchronization Rules
- Reports must consume module data; reports must not create parallel business logic.
- Printed patient documents must link to the source encounter/admission/order/invoice and write print history where backend supports it.
- Dashboard counts must use backend summaries or targeted queries, not client-side counting of huge datasets.
- Report runs should not block the UI; show progress and keep filters visible.

## Access and State Rules
- Reports, exports, audit logs, PHI logs, and dashboards are permission-gated.
- A user must not see a report category, button, patient context, export, or audit detail without access.
- After running/scheduling/exporting a report, update only the report run row, preview panel, schedule list, and notification badge.
- Audit log filters must be server-side where possible.

## Done Criteria
- All printable module outputs use one professional multi-page template.
- Reports are generated from data, not UI screenshots.
- Dashboards stay simple and link back to actual module work queues.
- Audit and PHI access logs are searchable, permission-aware, and export-safe.
- OPD/IPD turnaround time and bottleneck reports can be produced where data exists.
